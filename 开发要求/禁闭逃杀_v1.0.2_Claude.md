## 2. 必修问题一：自由阶段交易顺位卡严重 bug

### 2.1 当前问题

自由阶段交易顺位卡存在严重 bug：

- A 把顺位卡给 B 后，A 自己没有顺位卡；
- B 又把顺位卡给 C 后，C 可能拥有两张顺位卡；
- C 想再给别人时，只能把多张顺位卡一起给；
- 有时会出现很多张顺位卡集中在一个人手里；
- 玩家不能明确选择自己要交易哪一张；
- 顺位卡的唯一性和玩家行动顺序都会被破坏。

### 2.2 正确目标

顺位卡交易必须满足：

```text
每名存活玩家在每轮行动开始前必须且只能拥有 1 张顺位卡。
顺位卡本身必须是唯一实体，不能复制。
顺位卡交易应当是两名玩家之间交换 / 转移明确的一张卡，不能产生多张堆叠异常。
任意交易完成后，系统必须校验所有存活玩家的顺位卡状态。
```

### 2.3 建议实现方式

请重构顺位卡数据模型，不要把顺位卡只当作玩家字段里的字符串随便复制。

建议模型：

```ts
type TurnOrderCard = {
  id: string;
  order: number;
  ownerPlayerId: string | null;
  roundKey: string;
};
```

玩家可以保留便捷字段，但真实来源必须以 `TurnOrderCard.ownerPlayerId` 或统一的 order card ownership map 为准。

### 2.4 交易规则要求

请实现以下校验：

1. 玩家之间交易顺位卡时，必须明确选择交易对象和交易的具体顺位卡；
2. 如果规则设计为“交换顺位卡”，则 A 与 B 应互换各自顺位卡；
3. 如果当前实现支持“赠予顺位卡”，也必须保证赠予后不会让某人 0 张、某人 2 张；如无法保证，请限制为“交换顺位卡”；
4. 一次交易完成后，系统必须执行 `validateTurnOrderCards()`；
5. 若发现某名存活玩家不是恰好 1 张顺位卡，应阻止交易并回滚；
6. 顺位卡不能被复制，不能出现重复 owner，不能出现一张卡被多人持有；
7. 行动阶段开始后，禁止继续交易顺位卡。

### 2.5 验收标准

必须能验证：

- A 和 B 交换顺位卡后，A、B 仍然各 1 张；
- B 和 C 再交换后，三人仍各 1 张；
- 不会出现 C 持有 2 张或更多顺位卡；
- 不会出现 A 没有顺位卡；
- 玩家列表按最新顺位卡排序；
- 行动顺序使用最新顺位卡；
- 交易历史只在自由阶段有效。

## 3. 必修问题二：到达可抽卡房间后必须强制抽卡确认

### 3.1 当前问题

玩家到了可抽卡房间后，可能忘记抽卡，导致规则执行不完整。

### 3.2 正确目标

玩家到达可抽卡房间后，系统必须强制进入抽卡确认流程。

要求：

1. 如果玩家本轮最终停留在可抽卡房间，并且该房间本轮可抽卡，则行动结束前必须处理抽卡；
2. 可以设计为“必须抽卡”，或者至少“必须确认抽卡 / 放弃抽卡”；
3. 如果用户明确要求“强制抽卡”，则默认到达后自动抽取，不允许忘记；
4. 每个可抽卡房间通常每次行动只能抽一次；
5. 抽完后立即向当前玩家私密显示抽到了什么；
6. 不要只在超过负重时才显示抽卡结果；
7. 抽卡结果不能进入公开日志，除非规则要求公开；
8. 抽卡完成后，再处理是否超重、是否需要丢弃。

### 3.3 UI 要求

在玩家行动页中：

- 到达可抽卡房间后，显示“本房间可抽卡，系统已为你抽取 / 请确认抽卡”；
- 私密显示抽到的卡；
- 如果超重，进入丢弃流程；
- 未完成抽卡或丢弃前，不允许“结束行动”。

## 4. 必修问题三：优化公开日志与结算预览，合并公开日志和结算预览

### 4.1 当前问题

现在公开日志里有很多不该公开的信息，且结算预览和公开日志分裂，容易混乱。

### 4.2 正确目标

请把“公开日志”和“结算预览”合并为一套统一的结算展示系统。

核心原则：

```text
行动阶段不公开私人行动细节。
结算阶段才显示可以公开的信息。
不同玩家可以看到不同的私密结算信息。
房主可以看到完整裁判视图。
```

### 4.3 结算视图分层

请设计三类日志 / 结算信息：

#### A. Public Resolution Log 公开结算日志

所有玩家在回合结束后都能看到。

只包含规则允许公开的信息，例如：

- 哪些房间发生战斗 / 乱斗；
- 哪些房间发生暗影吸血；
- 哪个楼层成为毒气楼层；
- 火箭筒袭击了哪个房间；
- 玩家是否上交水粮；
- 哪些玩家成为暗影；
- 哪些暗影将在下一轮复活；
- 所有玩家当前生命值；
- 最终排名。

#### B. Private Resolution Log 私密结算日志

只有对应玩家能看到。

包含：

- 自己抽到了什么卡；
- 自己所在房间的私密房间效果；
- 自己因为某个隐藏房间状态无法抽卡；
- 自己使用果汁、道具、技能的私密结果；
- 自己被催眠后轮到自己时才知道的结果。

#### C. Host / Judge Resolution Log 房主裁判日志

房主可以看到完整信息，用于检查和纠错。

包括：

- 所有玩家私密行动；
- 所有抽卡结果；
- 所有房间效果；
- 所有隐藏技能和隐藏失败原因；
- 所有毒气投票明细；
- 所有结算过程。

### 4.4 房间效果的隐藏要求

房间效果不能一律公开。

示例：

```text
黑客锁了 B501 大仓库。
本轮去了 B501 大仓库的玩家发现不能抽卡。
只有这些玩家知道大仓库功能异常。
其他没去大仓库的人不知道。
公开日志不能写“黑客锁了大仓库”。
```

因此：

- 房间效果板块对每个玩家可能不同；
- 玩家只看到与自己相关的房间效果；
- 房主能看到完整房间效果；
- 公开日志只展示规则要求公开的信息。

### 4.5 合并方式

建议实现为一个统一组件：

```text
ResolutionReport
```

根据 viewer 类型展示不同内容：

```ts
type ResolutionViewer = 'public' | 'player' | 'host';
```

不要再维护多套互相矛盾的日志系统。

## 5. 必修问题四：毒气结果不要显示记票数量

### 5.1 当前问题

毒气结果显示了投票数量，暴露太多信息，也让界面复杂。

### 5.2 正确目标

公开显示时：

```text
只显示最终哪个楼层成为毒气楼层。

不要显示谁投了哪里。
```

例如公开日志只写：

```text
本轮毒气楼层：B4
```

如果多个并列：

```text
本轮毒气楼层：B3、B5
```

### 5.3 房主视图

房主裁判视图可以保留投票明细和票数，用于检查。

## 6. 必修问题五：除首页 / 玩家页外，其他位置优先显示角色名称，减少玩家 ID 干扰

### 6.1 当前问题

系统里大量显示玩家 ID，导致玩家看不懂，也影响推理体验。

### 6.2 正确目标

除首页、加入页、玩家个人身份页等必要位置外，游戏内大多数地方都应该显示“角色名称”，不要到处显示玩家 ID。

要求：

1. 行动顺序列表显示角色名称；
2. 结算日志显示角色名称；
3. 催眠师选择目标时显示角色名称，并可辅助显示昵称；
4. 私家侦探选择跟踪目标时显示角色名称，并可辅助显示昵称；
5. 交易界面显示角色名称；
6. 房主裁判视图可以显示角色名称 + 昵称 + ID，用于排查；
7. 技术 ID 只在调试 / 房主高级视图里显示。

建议统一封装：

```ts
formatPlayerName(player, viewerMode)
```

避免 UI 各处自己拼。

## 7. 必修问题六：轮次表达与水粮逻辑修复

### 7.1 当前问题

当前轮次表达和水粮结算有问题。

用户定义的正确表达：

```text
游戏一共 7 个行动轮次：首轮 + 第 1 轮 + 第 2 轮 + 第 3 轮 + 第 4 轮 + 第 5 轮 + 第 6 轮。
首轮是开局首轮，不算在正式 6 轮里。
首轮之后还有 6 轮正式轮次。
```

也就是 UI 应表达为：

```text
首轮
第 1 轮
第 2 轮
第 3 轮
第 4 轮
第 5 轮
第 6 轮
```

不要表达成“第 7 轮”。

### 7.2 水粮交付表达优化

首轮行动时，就要询问玩家是否为“第 1 轮”准备 / 上交水粮。

UI 文案建议改成：

```text
是否为下一轮交水粮？
```

不要只写“是否交水粮”，否则玩家不知道是为当前轮还是下一轮。

### 7.3 水粮结算显示要求

在结算阶段显示水粮结果。

要求：

1. 首轮时询问“是否为下一轮交水粮？”；
2. 到第 1 轮对应结算时，显示玩家此前是否交水粮，以及扣血结果；
3. 后续每轮行动中都询问是否为下一轮交水粮；
4. 玩家选择交水粮时，应明确消耗 1 水 + 1 粮食；
5. 如果缺水或缺粮，给出清晰提示；
6. 水粮结算日志按公开 / 私密规则展示；
7. 文案中不要出现“第 7 轮”。

### 7.4 注意

如果现有规则手册中的“从第 2 轮开始”与本次用户定义的“首轮 + 第 1~6 轮”表达存在冲突，请以本次 v1.0.2 用户定义为 UI 和程序执行标准，并在完成说明中写清楚你如何映射。

建议内部数据使用：

```ts
type RoundKey = 'initial' | 1 | 2 | 3 | 4 | 5 | 6;
```

显示函数：

```ts
formatRoundLabel(roundKey)
// initial -> 首轮
// 1 -> 第 1 轮
// ...
// 6 -> 第 6 轮
```

## 8. 必修问题七：果汁使用后必须弹出目标选择，下拉框数量与使用数量一致

### 8.1 当前问题

使用果汁之后，没有弹出“为谁使用”的目标下拉框，可能导致使用失败。

饮品师使用多个果汁时，虽然有使用数量，但不能按数量分别分配给多个玩家。

### 8.2 正确目标

使用果汁时必须进入目标分配流程。

要求：

1. 普通玩家使用 1 个果汁时，弹出 1 个目标选择框；
2. 饮品师选择使用 N 个果汁时，弹出 N 个目标选择框；
3. 每个目标选择框都可以选择一个玩家；
4. 可以允许多个果汁给同一个玩家，也可以给不同玩家；
5. 目标选项显示角色名称，必要时辅助显示昵称；
6. 未完成目标选择前，不允许确认使用；
7. 确认后按目标逐个结算果汁效果；
8. 结算日志要区分公开和私密信息；
9. 如果果汁数量不足，禁止确认。

### 8.3 饮品师特殊规则

饮品师：

- 果汁不占负重；
- 果汁可对其他玩家使用；
- 使用前可选 3 张效果卡再抽取。

请检查当前实现是否与规则手册一致，如果未完整实现，不要乱编，至少修复本次目标选择流程，并在 TODO 中列出剩余差异。

## 9. 必修问题八：本地热座模式添加随机生成玩家按钮

### 9.1 当前问题

本地热座模式每次都要手动定义角色和 ID，太麻烦。

### 9.2 正确目标

在本地热座 / 开发试玩模式中，添加快速随机生成按钮。

按钮建议：

```text
随机生成 9 名玩家
```

或：

```text
填充测试玩家
```

### 9.3 生成要求

点击后自动生成：

1. 9 名玩家；
2. 唯一昵称 / 测试名；
3. 唯一角色；
4. 初始生命值；
5. 合法初始基因点分配；
6. 合法出生房间；
7. 顺位卡按当前轮规则生成；
8. 必要的本地玩家身份；
9. 可直接开始游戏。

### 9.4 角色唯一性

随机生成时不能重复角色。

如果角色池超过 9 个，随机抽取 9 个不同角色。

### 9.5 使用限制

该按钮只用于：

- 本地热座模式；
- 开发测试模式；
- 房主调试模式。

不要在正式线上房间中默认暴露给普通玩家。

## 14. 鼠标验证流程

完成后，请按以下流程验证并在说明中写清楚：

### 14.1 顺位卡交易验证

1. 创建本地热座游戏；
2. 生成 9 名玩家；
3. 进入自由阶段；
4. A 与 B 交易顺位卡；
5. B 与 C 再交易顺位卡；
6. 检查 A / B / C 仍各 1 张顺位卡；
7. 检查没有任何玩家出现 0 张或 2 张以上顺位卡；

### 14.2 抽卡验证

1. 让玩家到达可抽卡房间；
2. 系统强制进入抽卡确认 / 自动抽卡流程；
3. 抽完立即显示抽到的卡；
4. 结束行动前不能跳过必要抽卡处理；
5. 本次行动不能重复抽同一房间。

### 14.3 日志验证
