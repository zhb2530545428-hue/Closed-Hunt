# 四、具体修复要求

## 1. 首轮规则修正：出生房间选择后立即结算第一次战斗

### 当前问题

之前把“首轮”理解错了。现在需要修正为：

```text
选角色 / 分配属性 / 选择出生房间
→ 所有人出生房间确定
→ 立即进行第一次战斗结算
→ 然后进入正式第 1 轮
```

### 新规则要求

新增或调整游戏流程：

```text
LOBBY / SETUP
→ 角色与属性确认
→ 出生房间选择
→ 首轮出生战斗结算
→ 第 1 轮自由阶段
→ 第 1 轮行动阶段
→ 第 1 轮结算阶段
...
→ 第 6 轮结算阶段
→ 最终排名
```

注意：

- “首轮”不是正式第 1 轮。
- 首轮只用于出生房间后的第一次战斗结算。
- 首轮没有移动、抽卡、房间功能、交易、毒气投票、水粮、职业主动技能、火箭筒、房间效果等其他功能。
- 首轮只结算出生房间内的战斗 / 乱斗。
- 若同一出生房间有 2 名或以上存活玩家，按正常战斗 / 乱斗规则结算。
- 若出生房间只有 1 人，不发生战斗。
- 首轮战斗伤害是否导致暗影，按当前项目死亡 / 暗影规则执行，但必须注意不要触发正式轮次的其他结算项目。
- 首轮结算日志需要脱敏：公开只显示哪些出生房间发生了战斗 / 乱斗，不显示谁在房间里、谁打谁、谁扣几点血。
- 首轮结算结束后，统一公开所有玩家当前生命值，然后进入正式第 1 轮。

### UI 要求

在选择出生房间完成后，应出现明确步骤：

```text
首轮出生战斗结算
```

房主 / 本地热座模式可以点击确认执行。

执行后显示：

```text
首轮结算完成，进入第 1 轮
```

不要再把首轮显示成“第 0 轮”或“第 7 轮”。

## 2. 催眠师强制移动修复

### 当前问题

被催眠师催眠的人，有时仍然需要自己点击前往哪个房间，或者界面仍然允许他选择其他房间。这是错误的。

### 正确规则

催眠师在自己的行动阶段使用技能：

1. 选择 1 名存活玩家作为催眠目标。
2. 指定 1 个目标房间。
3. 系统判断：从被催眠玩家当前房间到指定房间，是否可以在 5 步之内到达。
4. 注意：这里使用固定 5 步判断，和被催眠玩家自己的速度无关。
5. 如果 5 步之内无法到达，催眠失败。
6. 如果 5 步之内可以到达，催眠成功。
7. 催眠成功后，被催眠玩家轮到自己行动时：
   - 系统直接将他的移动目标锁定为催眠师指定房间；
   - 不允许他自己选择其他房间；
   - 其他房间不可点击或禁用；
   - 界面必须有明显特殊标记，告诉该玩家“你已被催眠，本轮将被强制移动到 X 房间”；
   - 被催眠玩家仍然可以在抵达后处理该房间允许的私密信息与后续必要动作，除非规则另有限制。

### 信息隐藏要求

- 催眠师催眠成功后，目标玩家不能立即知道。
- 目标玩家只有轮到自己行动时，才看到“已被催眠”的提示。
- 其他玩家不能看到谁被催眠、被催眠到哪里。
- 公开日志不得暴露催眠目标和目标房间。

### 代码要求

请建立明确的 pending hypnosis 状态，例如：

```ts
type PendingHypnosis = {
  hypnotistPlayerId: string;
  targetPlayerId: string;
  forcedRoomId: string;
  roundId: string;
  status: 'pending' | 'failed' | 'applied' | 'expired';
};
```

或使用项目现有结构，但必须保证：

- 催眠状态只在本轮有效；
- 每名催眠师每回合最多催眠 1 人；
- 被催眠目标轮到自己行动时自动应用；
- 应用后状态不可重复触发；
- 回合结束时清理未处理或过期催眠状态；
- 不要因为刷新页面导致催眠状态丢失。

### 验收案例

至少验证：

1. 被催眠目标 5 步内可达，成功强制移动。
2. 被催眠目标 5 步内不可达，催眠失败。
3. 催眠成功后，目标玩家行动页不能选择其他房间。
4. 催眠目标轮到自己前，不显示被催眠信息。
5. 其他玩家视角看不到催眠细节。
6. 催眠状态不会重复执行。

## 3. 本地热座模式增加“重置”按钮

### 当前问题

本地热座模式整局结束后，没有办法方便地重新开一局测试。

### 要求

在本地热座 / 测试模式中，增加一个明确按钮：

```text
重置
```

要求：

- 任何时间都可以点，不用等整局结束。
- 点击后需要二次确认，避免误触。
- 确认后清空当前对局状态，回到选人 / 开局界面。
- 清空内容包括：
  - 玩家状态；
  - 角色分配；
  - 出生房间；
  - 当前轮次；
  - 行动状态；
  - 交易状态；
  - 道具库存；
  - 房间库存；
  - 毒气楼层；
  - 日志；
  - 暗影状态；
  - 催眠状态；
  - 黑客关闭房间状态；
  - 所有临时结算状态。
- 地图编辑器配置、项目基础规则配置不要被重置。

### UI 要求

按钮可以放在：

```text
本地热座控制区 / 房主控制台 / 调试区域
```

但不要放得太隐蔽。

按钮文案建议：

```text
重置本局
```

确认弹窗：

```text
确定要重置当前对局吗？这会清空本局所有玩家、轮次、道具和日志，并回到选人界面。
```

## 4. 水、粮、药、果汁、肾上腺素等资源使用后移到结算阶段

### 当前问题

之前把水、粮、药、果汁、肾上腺素等按钮放在行动阶段末尾。但玩家通常需要看到本轮结算预览、自己生命值和当前风险后，才会决定是否使用这些物资。

所以这些资源不应该在行动阶段使用，而应该放到结算阶段。

### 新规则要求

以下资源使用入口后移到结算阶段：

```text
水
粮
药
果汁
肾上腺素
```

注意：

- 行动阶段不再显示这些资源的使用按钮。
- 行动阶段仍然可以显示背包内容，但不要让玩家在行动阶段提交这些资源的使用选择。
- 结算阶段应提供一个“玩家资源选择 / 结算准备”步骤。
- 玩家在该步骤中看到自己的当前公开/私密必要信息后，再选择是否使用水、粮、药、果汁、肾上腺素。
- 这些选择应在对应结算步骤中生效。

### 结算阶段建议流程

请检查并优化结算阶段流程。建议拆成：

```text
1. 结算预览生成：只生成本轮将要发生的公开事件类别，不提前暴露隐藏信息
2. 玩家资源选择：各玩家私密选择是否使用水、粮、药、果汁、肾上腺素
3. 房主确认 / 系统应用结算
4. 按固定顺序执行结算
5. 输出脱敏公开日志与玩家私密日志
6. 统一公开所有玩家最终生命值
```

注意：

- 水粮属于对应水粮步骤。
- 药、果汁、肾上腺素属于道具回血 / 状态步骤。
- 肾上腺素的“下一轮生效”语义必须正确。
- 如果当前规则或代码已经有“为下一轮使用”的表述，请统一检查是否和结算阶段后移冲突。

### UI 要求

结算阶段玩家私密面板中显示：

```text
本轮结算资源选择
水：当前数量 X，是否上交
粮：当前数量 X，是否上交
药：当前数量 X，是否使用
果汁：当前数量 X，是否使用 / 使用数量 / 目标选择
肾上腺素：当前数量 X，是否使用
```

数量为 0 时仍显示，但按钮禁用。

注意：果汁目标选择之前已经修过，本次不要无意义重构，只需要把入口移动到结算阶段，并保证现有目标选择逻辑仍可用。

### 信息隐藏要求

- 每名玩家只能看到自己的资源选择。
- 其他玩家不能看到谁准备用水、粮、药、果汁、肾上腺素。
- 公开日志只在结算完成后显示必要公开结果。
- 不要通过某个玩家是否上交水粮，提前暴露其所在房间或状态。

### 验收案例

至少验证：

1. 行动阶段不再出现水 / 粮 / 药 / 果汁 / 肾上腺素使用按钮。
2. 结算阶段出现资源选择面板。
3. 数量为 0 的按钮显示但禁用。
4. 玩家可在结算阶段选择上交水粮。
5. 玩家可在结算阶段使用药。
6. 玩家可在结算阶段使用果汁，并保留已修好的目标选择能力。
7. 玩家可在结算阶段使用肾上腺素，且效果进入下一轮。
8. 资源选择不会泄露给其他玩家。
