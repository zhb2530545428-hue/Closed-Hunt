# 《禁闭逃杀》电子版 v0.3 开发指令

## 一、当前项目背景

你正在继续开发《禁闭逃杀》电子版项目。

当前仓库：

```text
https://github.com/zhb2530545428-hue/Closed-Hunt.git
```

项目根目录中应当已经存在：

```text
禁闭逃杀_规则手册.md
禁闭逃杀_地图.png
```

前置版本假设：

- v0.1 已完成：房间创建、9 人大厅、玩家加入、职业/基因/出生点、轮次阶段、行动提交、毒气投票、公共战况页、房主控制台。
- v0.2 已完成：地图展示、道具库存、房间库存、抽卡、负重、战斗/乱斗/枪械压制、水粮、毒气、暗影、火箭筒、基础道具效果、房主结算预览。

本次开发版本为：

```text
v0.3：地图移动合法性与行动体验升级
```

本次目标不是重做项目，而是在现有 v0.2 基础上向前推进。

---

## 二、本次 v0.3 的核心目标

v0.3 的核心目标是：

```text
把玩家行动从“随便填写目标房间”升级为“系统根据地图规则判断能不能移动”。
```

也就是说，系统需要知道：

- 玩家当前在哪个房间；
- 玩家当前速度是多少；
- 哪些房间在本轮可到达；
- 哪些房间超出速度范围；
- 选择目标房间后，系统推荐一条最短路径；
- 路径经过 102 激光室时自动提示并结算伤害；
- 特殊移动规则如何影响可达范围；
- 玩家不能原地停留；
- 行动日志中记录移动路径和触发效果。

v0.3 完成后，游戏应该从“电子记录/结算工具”进一步变成“真正能约束行动的电子桌游”。

---

## 三、开发总原则

请严格遵守以下原则：

1. 规则来源只以 `禁闭逃杀_规则手册.md` 为准，不要自己编规则。
2. 不要重构掉 v0.1 / v0.2 已经稳定工作的功能。
3. 不要为了地图移动系统重写整个游戏状态管理。
4. 不要把地图规则硬编码在 UI 组件中。
5. 地图移动规则必须尽量放在独立的配置和工具函数中。
6. 先做稳定、可验证的“房间按钮 + 路径预览”方案，不要追求复杂拖拽地图。
7. 地图图片继续使用项目内已有的 `禁闭逃杀_地图.png`，不要重新绘制地图。
8. 本版本不做真实多人后端，不做账号登录，不做匹配系统。
9. 本版本不做完整职业技能自动化。
10. 本版本不做精确地图热区点击。

### 2. 路径搜索工具函数

新增或完善路径搜索工具函数，例如：

```text
src/game/utils/movement.ts
```

需要实现以下能力：

```ts
getReachableRooms(params)
findShortestPath(params)
validateMove(params)
getMovePreview(params)
```

可根据项目结构调整命名，但功能必须具备。

#### 2.1 getReachableRooms

输入：

```ts
{
  fromRoomId: string;
  speed: number;
  playerStatus: 'alive' | 'shadow';
  inventory: Item[];
  activeStatuses: PlayerStatus[];
  roundState: GameRoundState;
}
```

输出：

```ts
{
  roomId: string;
  distance: number;
  path: string[];
  moveType: 'normal' | 'rope' | 'shadow' | 'helicopter' | 'trash_chute' | 'portal';
  warnings: string[];
}[]
```

不要求完全照抄这个类型，但必须表达：

- 可达房间；
- 消耗步数；
- 推荐路径；
- 是否触发特殊移动；
- 是否有风险提示。

#### 2.2 findShortestPath

根据地图图结构计算从 A 到 B 的最短路径。

基础要求：

- 普通玩家只能沿相邻房间移动；
- 每移动到相邻房间消耗 1 步；
- 上下楼必须通过地图中的楼梯或规则允许的特殊方式；
- 左右楼栋之间必须通过地图中的廊桥或特殊移动方式；
- 未连接房间不能直接移动。

#### 2.3 validateMove

校验玩家本轮移动是否合法。

必须检查：

- 起点是否为玩家当前位置；
- 目标房间是否存在；
- 目标房间是否等于当前位置；
- 是否在当前速度范围内；
- 是否符合暗影 / 绳索 / 直升机 / 回收站 / 传送室等特殊移动规则；
- 若不合法，给出明确错误原因。

#### 2.4 getMovePreview

在玩家确认移动前，生成移动预览。

预览内容至少包括：

```text
目标房间
预计消耗步数
推荐路径
是否经过 102 激光室
是否会触发激光伤害
是否经过 B105 回收站
是否经过 B403 传送室
是否使用直升机
是否使用绳索
是否为暗影特殊移动
```

### 4. 特殊移动规则

v0.3 至少要实现以下特殊移动逻辑。

#### 4.1 102 激光室

规则：

```text
存活玩家经过或停留 102 激光室，立即扣 1 点生命值；暗影不受影响。
```

实现要求：

- 如果移动路径经过 102，行动预览中必须提示；
- 玩家确认移动后，系统立即扣除 1 点生命值；
- 如果玩家最终停留在 102，也视为触发；
- 暗影经过或停留 102 不扣血；
- 日志中记录激光室伤害。

注意：生命值最低为 0；死亡检查仍应遵守结算阶段最后统一检查的规则，不要在移动中直接把玩家变暗影，除非现有 v0.2 已经有统一处理机制并能兼容。

#### 4.2 绳索

规则：

```text
持有绳索的玩家上下楼时可以不经过楼梯，但每跨 1 层仍消耗 1 步。
```

实现要求：

- 持有绳索的存活玩家，可使用绳索上下楼；
- 每跨 1 层消耗 1 步；
- 使用绳索不能无视水平房间移动；
- 行动预览中标记“使用绳索”；
- 日志中记录使用绳索移动。

如果完整楼层坐标关系难以一次实现，允许先实现一个保守版本：

- 只允许在同一竖向楼栋或地图配置明确允许的房间之间使用绳索；
- 不允许任意房间跨楼传送。

不要把绳索实现成免费传送。

#### 4.3 暗影移动

规则：

```text
暗影不需要通过楼梯即可上下楼。
```

实现要求：

- 暗影仍然继承死亡时速度；
- 暗影上下楼不依赖楼梯；
- 暗影移动仍应消耗步数；
- 暗影不能使用房间功能，不能抽取道具，不能毒气投票；
- 暗影经过激光室不受伤害；
- 行动预览中标记“暗影移动”。

#### 4.4 B105 回收站 → B503 垃圾场

规则：

```text
玩家经过 B105 回收站时，可消耗 1 步通过垃圾管道单向滑行至 B503 垃圾场。
```

实现要求：

- 当路径到达或经过 B105 时，系统允许选择使用垃圾管道；
- 使用后目标可以到达 B503；
- 额外消耗 1 步；
- 单向：B503 不能反向滑回 B105；
- 行动预览中显示是否使用垃圾管道；
- 日志中记录该特殊移动。

#### 4.5 B403 传送室

规则：

```text
玩家经过 B403 传送室时，可消耗 1 步传送至任意普通房间。
```

实现要求：

- 玩家能够到达或经过 B403 时，可选择使用传送；
- 使用传送额外消耗 1 步；
- 传送目标必须是普通房间；
- 不能传送到功能房间；
- 行动预览中显示传送目标；
- 日志中记录传送。

普通房间判定请基于规则配置，不要硬编码在 UI。

#### 4.6 202 停机坪直升机

规则：

```text
玩家在 202 停机坪后，下一轮可消耗 1 步搭乘直升机前往指定房间。
```

直升机可到达房间：

```text
B101
101
B103
201
B105
```

实现要求：

- 玩家上一轮或当前状态中需要记录是否拥有“下一轮直升机移动资格”；
- 有资格时，本轮行动页展示直升机移动选项；
- 使用直升机消耗 1 步；
- 目标只能是上面列出的 5 个房间；
- 使用后清除资格；
- 日志中记录直升机移动；
- 如果玩家没有资格，不显示或禁用该选项。

#### 4.7 肾上腺素速度

规则：

```text
肾上腺素从下一轮开始生效，生效轮速度变为 10。
```

实现要求：

- 如果 v0.2 已经实现肾上腺素状态，则 v0.3 的可达范围计算必须读取该状态；
- 生效轮行动页显示速度为 10；
- 可达房间按速度 10 计算；
- 结算结束后恢复原速度。

## 六、推荐开发顺序

请按下面顺序开发，降低出错概率。

### Step 1：梳理现有结构

先阅读现有代码，确认：

- 游戏状态存在哪里；
- 玩家数据结构；
- 房间配置；
- 道具配置；
- 行动提交结构；
- 地图图片展示位置；
- 日志系统；
- 结算流程。

不要一开始就写新代码。

### Step 2：建立 mapGraph 配置

根据规则手册和地图图片录入房间节点与连接关系。

完成后写基础校验：

- 所有规则房间都存在；
- 不存在重复房间 ID；
- 所有邻接房间 ID 都能找到；
- 邻接关系尽量双向，特殊单向移动单独配置。

### Step 3：实现 movement 工具函数

先用纯函数实现移动能力，不接 UI。

建议至少包含：

```text
getRoomById
getNeighbors
findShortestPath
getReachableRooms
validateMove
getMovePreview
```

### Step 4：写轻量测试或调试页面

如果项目已有测试框架，可以写单元测试。

如果没有测试框架，至少写一个开发期调试输出或简单验证函数，确保：

- B103 到相邻房间可达；
- 超出速度不可达；
- 原地停留不可提交；
- 经过 102 会提示激光伤害；
- B105 可以单向到 B503；
- B403 可以传送到普通房间；
- 直升机只能到指定 5 个房间。

### Step 5：升级玩家行动页

接入可达房间计算。

玩家不再随便填终点，而是从系统生成的可达房间中选择。

### Step 6：接入特殊移动和行动提交

把特殊移动、路径、触发效果写入行动提交结构。

### Step 7：接入日志和状态变化

确认移动后：

- 更新玩家位置；
- 应用激光室伤害；
- 设置或清除直升机资格；
- 写入日志；
- 保留毒气投票和房间行动选择。

### Step 8：回归 v0.2 结算

确认 v0.3 没有破坏：

- 道具库存；
- 战斗结算；
- 毒气；
- 水粮；
- 暗影；
- 火箭筒；
- 最终排名。

## 七、验收标准

完成后请确保可以用鼠标完成以下验证。

### 基础移动验证

```text
1. 创建房间。
2. 填满或模拟 9 名玩家。
3. 设置玩家出生点和速度。
4. 开始游戏。
5. 进入某个玩家行动页。
6. 页面显示当前位置、当前速度、可达房间列表。
7. 选择速度范围内目标房间，可以提交。
8. 选择速度范围外目标房间，系统禁止提交并说明原因。
9. 选择当前位置作为目标，系统禁止提交。
10. 提交后玩家位置更新，日志记录路径。
```

### 激光室验证

```text
1. 让玩家路径经过或停留 102。
2. 行动预览提示会受到激光伤害。
3. 确认移动后生命值 -1。
4. 日志记录激光室伤害。
5. 暗影经过 102 不扣血。
```

### 绳索验证

```text
1. 给玩家添加绳索。
2. 查看可达房间是否因为绳索发生变化。
3. 使用绳索上下楼时，预览显示使用绳索。
4. 每跨 1 层消耗 1 步。
```
