## 1. 角色选择不能重复：隐藏选择 + 撞车统一抽取

### 当前问题

多人可以选择相同角色，且第一次选择角色时互相可见，破坏公平性。

### 正确规则

1. 准备阶段，玩家第一次选择角色时，彼此不可见。
2. 所有玩家提交角色选择后，再统一解析。
3. 如果某个角色只有 1 名玩家选择，则该玩家获得该角色。
4. 如果某个角色被 2 名或以上玩家选择，这些玩家全部进入“撞车池”。
5. 所有撞车玩家必须等所有玩家选完后，一起从剩余未分配角色中随机抽取。
6. 如果有多个角色同时撞车，所有撞车玩家统一进入同一个抽取池，而不是每组单独抽。
7. 抽取必须无放回，保证最终所有玩家角色唯一。
8. 最终角色分配完成后，进入游戏时角色可以公开显示。

### 推荐实现

新增或调整角色分配逻辑：

```ts
type RoleChoice = {
  playerId: string
  preferredRoleId: string
  submittedAt: number
}

type RoleResolutionResult = {
  lockedAssignments: Record<PlayerId, RoleId>
  collisionPlayerIds: PlayerId[]
  availableRoleIds: RoleId[]
  finalAssignments: Record<PlayerId, RoleId>
}
```

解析算法：

```text
1. 收集所有玩家 preferredRoleId。
2. 按 roleId 分组。
3. group.size === 1：该玩家锁定该角色。
4. group.size > 1：该组所有玩家进入 collisionPlayerIds。
5. availableRoleIds = 全部角色 - 已被唯一锁定的角色。
6. collisionPlayerIds 随机排序。
7. availableRoleIds 随机排序。
8. 逐个为 collisionPlayerIds 分配 availableRoleIds。
9. 写入最终 roleId。
```

注意：

- 撞车角色本身如果没有被唯一锁定，也属于剩余可抽角色。
- 准备阶段 UI 只能显示“已提交 / 未提交”，不能显示别人选了什么角色。
- 房主也不应在角色最终解析前看到所有人的选择，除非项目已有主持人全知模式；默认不要暴露。
- 需要添加测试：
  - 9 人全部选择不同角色；
  - 2 人撞同一角色；
  - 多组同时撞车；
  - 9 人全部撞同一角色；
  - 角色池数量大于 9 时，撞车玩家从剩余角色中抽取。

## 2. 地图步数计算精修：坐标 + 图结构 + 校验工具

### 当前问题

部分地图步数计算有问题。不能只靠手写邻接或视觉估算，必须建立更精细的地图数据。

### 核心原则

地图系统必须拆成两层：

```text
A. 规则层：mapGraph，负责路径、步数、连接、特殊移动，是唯一计算来源。
B. 展示层：mapCoords，负责地图图片上的坐标、房间热区和高亮，不参与规则判断。
```

不要直接用图片像素距离计算步数。图片只负责展示，步数必须由图结构边计算。

### 必做文件建议

```text
src/game/config/mapGraph.ts
src/game/config/mapCoords.ts
src/game/logic/movement.ts
src/game/logic/movement.test.ts
src/app/dev/map-checker/page.tsx        // 如果项目使用 app router
或 src/pages/dev/map-checker.tsx        // 如果项目使用 pages router
```

### mapGraph 要包含

```ts
type RoomNode = {
  id: string              // 如 "B503"
  floor: string           // 如 "B5"
  name?: string           // 如 "垃圾场"
  building?: 'left' | 'right' | 'center' | 'unknown'
  kind: 'normal' | 'function' | 'start' | 'special'
  isOrdinaryRoom: boolean
}

type MapEdge = {
  from: string
  to: string
  cost: number            // 通常为 1
  bidirectional: boolean
  type: 'adjacent' | 'stairs' | 'bridge' | 'shortcut' | 'helicopter' | 'shadow' | 'rope'
  enabledBy?: 'normal' | 'rope' | 'shadow' | 'helicopter' | 'recycleTube' | 'teleport'
  note?: string
}
```

### mapCoords 要包含

使用归一化坐标，不要写死像素：

```ts
type RoomCoord = {
  roomId: string
  x: number      // 0 到 1，相对于地图图片宽度
  y: number      // 0 到 1，相对于地图图片高度
  w: number      // 0 到 1
  h: number      // 0 到 1
  cx: number     // 中心点 x，可由 x + w/2 生成
  cy: number     // 中心点 y，可由 y + h/2 生成
}
```

### 地图精修流程

请实现一个开发辅助页 `/dev/map-checker`：

1. 显示 `禁闭逃杀_地图.png`。
2. 在图片上叠加每个房间的坐标框。
3. 显示房间 ID 和名称。
4. 点击任意两个房间，显示：
   - 最短路径；
   - 总步数；
   - 经过房间；
   - 是否经过 102 激光室；
   - 是否使用楼梯 / 廊桥 / 特殊移动。
5. 支持切换玩家状态：
   - 普通玩家；
   - 持有绳索；
   - 暗影；
   - 肾上腺素生效；
   - 可使用停机坪直升机；
   - 可使用回收站滑行；
   - 可使用传送室。
6. 对配置不确定的边，标红并写入 `MAP_REVIEW.md`。

### 重要要求

- 不要让 Claude 凭空重画地图。
- 不要用图片距离自动推断房间可达性。
- 所有可通行连接必须进入 `mapGraph.ts`。
- 所有房间视觉位置必须进入 `mapCoords.ts`。
- 地图展示、玩家行动页、房主页面都复用同一份坐标配置。
- 如果看不清某个连接，不要擅自猜，写入 `MAP_REVIEW.md`，列出“需要人工确认的连接”。

### 必须补测试

至少覆盖：

```text
1. 普通玩家不能原地停留。
2. 普通玩家速度不足时不能到达。
3. 普通玩家上下楼必须通过楼梯。
4. 持有绳索时，上下楼可不经过楼梯，但每跨 1 层仍消耗 1 步。
5. 暗影不需要楼梯即可上下楼。
6. B105 回收站可以单向滑行至 B503 垃圾场。
7. B403 传送室可以传送至任意普通房间。
8. 202 停机坪下一轮可消耗 1 步飞往规则指定房间。
9. 经过 102 激光室时，只记录待结算伤害，不立即公开扣血。
10. 肾上腺素生效轮速度为 10。
```

## 3. 抽卡房间不能无限抽卡，抽完后立即显示结果

### 当前问题

可以抽卡的房间里，玩家可能无限次抽卡；并且抽完后没有立即显示抽到了什么，只有超重时才显示。

### 正确规则

1. 通常情况下，每名玩家每次行动到达一个抽卡房间后，只能执行一次抽卡。
2. 抽卡完成后，应立即向该玩家私密显示抽取结果。
3. 抽卡结果不能因为未超重而不显示。
4. 如果超重，先显示抽到了什么，再进入超重处理。
5. 抽取结果属于玩家私密信息，不应立即写入公共日志。

### 推荐状态

```ts
type TurnActionState = {
  playerId: string
  round: number
  hasUsedRoomFunction: boolean
  hasDrawnFromRoom: boolean
  drawnRoomId?: string
  privateDrawResult?: Item[]
}
```

### UI 要求

- 抽卡按钮点击后立即禁用。
- 显示“本次抽到：水、粮食、刀……”等私密结果。
- 超重时显示“当前负重 X / 上限 Y，请丢弃到不超过负重”。
- 下一名玩家开始行动时，上一名玩家的抽卡结果必须清空，不得残留。

### 当前问题

项目中仍存在“酒”的命名、道具、UI、日志或代码标识。

把所有面向玩家的“酒”统一替换为“果汁”。

请全文搜索：

```text
酒
jiu
wine
alcohol
liquor
beer
drink
```

### 替换要求

## 5. 饮品师：多个果汁要能分配给多个目标

### 当前问题

饮品师可以选择使用果汁数量，但不能按数量分配给多个人。

### 正确规则

1. 饮品师有几个果汁，就可以选择使用几个。
2. 使用数量为 N 时，界面必须生成 N 个目标选择条。
3. 每个果汁都要有独立目标。
4. 目标可以是同一个人，也可以是不同人，除非规则手册另有禁止。
5. 饮品师的果汁可对其他玩家使用。
6. 如果当前实现支持果汁效果卡选择，效果卡选择也要和每个果汁实例绑定，不要共用一个目标。

### 推荐数据结构

```ts
type DrinkUseAssignment = {
  instanceId: string
  targetPlayerId: string
  selectedEffectCardIds?: string[]
}

type DrinkAction = {
  playerId: string
  quantity: number
  assignments: DrinkUseAssignment[]
}
```

### 校验

- `assignments.length === quantity`
- 每个 assignment 都必须有 targetPlayerId。
- quantity 不能超过饮品师当前持有果汁数量。
- 提交后扣除对应数量果汁。
- 目标效果进入结算队列，不要提前公开。

## 7. 严格按顺位行动，结束行动后不能更改

### 当前问题

玩家可能不按顺序行动，或者结束行动后还能修改。这样会影响抢资源、先后进入房间拿装备等核心规则。

### 正确规则

1. 每轮抽顺位卡后，玩家必须严格按顺位依次行动。
2. 同一时间只有当前顺位玩家可以操作。
3. 非当前玩家只能查看允许查看的信息，不能提交行动。
4. 当前玩家点击“结束行动 / 确认行动”后，本轮行动锁定，不能修改。
5. 抽卡、拿资源、使用房间功能等与先后顺序有关的效果，必须在当前玩家行动确认后立即写入私密状态和库存状态。
6. 伤害与公共日志不能立即公开，见第 8 和第 13 节。

### 推荐状态

```ts
type RoundTurnState = {
  round: number
  order: PlayerId[]
  currentIndex: number
  currentPlayerId: PlayerId
  actedPlayerIds: PlayerId[]
  lockedActions: Record<PlayerId, LockedAction>
}
```

### UI 要求

- 当前行动玩家高亮。
- 其他玩家看到“等待 P? 行动”。
- 已行动玩家显示“已锁定，不可更改”。
- 房主不能随意代改已锁定行动，除非进入“纠错模式”，并写入管理员日志。

## 8. 所有伤害延迟到回合结束阶段依次结算

### 当前问题

激光室等伤害被立即扣血，后续行动玩家可能通过生命变化推断掉血玩家经过了哪里，破坏隐藏移动。

### 正确规则

1. 行动阶段可以记录“待结算伤害事件”。
2. 不能立即公开扣血。
3. 不能立即更新公共生命值。
4. 所有伤害必须等到整个回合进入结算阶段后，按规则手册的固定顺序依次结算。
5. 结算完成后，再统一公开生命变化和日志。

### 需要延迟的事件

至少包括：

```text
激光室伤害
战斗 / 乱斗伤害
暗影吸血
火箭筒伤害
毒气伤害
水粮不足伤害
病毒携带者伤害
其他职业或道具造成的伤害
```

### 推荐数据结构

```ts
type PendingEffect = {
  id: string
  round: number
  phase: 'action' | 'resolution'
  timing: 'roomEffect' | 'battle' | 'shadow' | 'rocket' | 'gas' | 'hunger' | 'item' | 'status' | 'deathCheck'
  visibility: 'private' | 'host' | 'publicAfterResolution'
  sourcePlayerId?: string
  targetPlayerIds: string[]
  roomId?: string
  amount?: number
  payload?: unknown
}
```

### 特别说明：激光室

- 经过或停留 102 激光室时，不要立即扣公共生命。
- 可以在该玩家私密界面提示：“本轮路径包含激光室，结算阶段将受到 -1 生命。”
- 但不要让其他玩家看到生命变化或日志。
- 结算阶段“房间效果”步骤统一处理。

## 9. 催眠师技能修复

### 当前问题

催眠师使用技能后，被催眠玩家可能立即知道；催眠师每回合使用次数不严格；强制移动逻辑不准确；被催眠玩家仍能自己选移动地点。

### 正确规则

1. 催眠师每回合最多使用 1 次催眠。
2. 每次只能催眠 1 名存活玩家。
3. 催眠师选择目标和目标房间后，被催眠玩家不能立即知道。
4. 被催眠玩家只有轮到自己行动时，才发现自己已经被催眠。
5. 催眠移动按 5 步内判断，和被催眠玩家本人的速度无关。
6. 催眠移动不能使用捷径。
7. 如果 5 步内到不了目标房间，则催眠失败，被催眠玩家恢复自由行动。
8. 如果催眠成功，被催眠玩家强制前往该房间，不允许自己再选择移动地点。
9. 强制移动成功后，被催眠玩家仍可正常触发终点房间功能、抽卡，并提交毒气投票。
10. 每名玩家整场只能被催眠 1 次。

### 推荐状态

```ts
type HypnosisState = {
  round: number
  hypnotistPlayerId: string
  targetPlayerId: string
  forcedRoomId: string
  status: 'pendingHidden' | 'revealedToTarget' | 'succeeded' | 'failed'
}
```

### UI 要求

催眠师行动时：

- 目标列表显示玩家 ID + 昵称 + 角色名。
- 只能选择存活玩家。
- 不能选择已经被催眠过的玩家。
- 本回合已使用过催眠后禁用按钮。

被催眠玩家行动时：

- 打开行动面板时才显示：“你本轮被催眠，系统将尝试强制你移动至指定房间。”
- 如果 5 步内可达：显示强制路径和目标房间，禁用普通目标选择。
- 如果 5 步内不可达：显示催眠失败，可自由行动。

日志要求：

- 行动阶段不公开催眠目标和目标房间。
- 结算或回合公开日志中只公开规则允许公开的信息。

```ts
type PrivateActionDraftKey = `${roomId}:${round}:${playerId}`
```

## 11. 私家侦探跟踪规则修复

### 当前问题

私家侦探跟踪可能显示目标要去的房间；目标范围不正确；发动技能后仍要选移动房间。

### 正确规则

1. 私家侦探行动时，在“普通移动”和“跟踪”之间二选一。
2. 选择跟踪后，不再选择目标房间。
3. 私家侦探只能跟踪本轮顺位在自己之前、且已经完成行动的玩家。
4. 跟踪目标列表不能显示目标去了哪个房间。
5. 私家侦探确认跟踪后，系统把他移动到目标玩家本轮最终房间。
6. 私家侦探可正常触发该房间效果和抽卡。
7. 私家侦探整场只能使用 3 次，每名玩家只能被跟踪 1 次。
8. 被催眠时无法使用跟踪技能。

### UI 要求

跟踪目标列表只显示：

```text
P2 李四｜雇佣兵｜已行动
P4 王五｜病毒携带者｜已行动
```

不能显示：

```text
目标房间 B107
目标路径
目标抽卡结果
目标毒气投票
```

确认跟踪后，私家侦探自己的私密界面可以看到自己最终所在房间，以便使用房间效果。

## 13. 公开日志必须延迟到回合结束后展示

### 当前问题

每动一个人，下一个人就能看到公开日志，导致行动信息泄露。

### 正确规则

1. 行动阶段不应实时公开移动、抽卡、技能目标、毒气票、伤害等日志。
2. 行动阶段只能显示极少量非敏感信息，例如：
   - 当前轮次；
   - 当前顺位；
   - 谁已完成行动。
3. 具体行动日志进入 `privateLogs` 或 `roundLogBuffer`。
4. 到结算阶段结束后，按规则允许公开的内容统一写入 `publicLogs`。
5. 公共日志应对应规则手册中的公开信息范围。

### 推荐日志结构

```ts
type GameLog = {
  id: string
  round: number
  phase: string
  visibility: 'private' | 'hostOnly' | 'publicAfterResolution' | 'public'
  playerId?: string
  message: string
  payload?: unknown
}
```

行动阶段：

```text
private：你抽到了 1 把刀。
hostOnly：P3 已提交行动。
public：P3 已完成行动。
```

结算结束后：

```text
public：B107 发生战斗。
public：第 2 轮毒气楼层为 B4。
public：P5 成为暗影。
public：所有玩家当前生命值……
```
