# 《禁闭逃杀》地图编辑器开发指令｜精简版

## 0. 当前任务定位

你现在继续开发《禁闭逃杀》电子版项目。

当前仓库：

```text
https://github.com/zhb2530545428-hue/Closed-Hunt.git
```

项目根目录里已有：

```text
禁闭逃杀_规则手册.md
禁闭逃杀_地图.png
```

本次任务不是继续做战斗、职业、联机、结算或正式游戏 UI，而是单独实现一个轻量的：

```text
根据原地图重建一版结构化地图草稿，并提供一个开发用地图编辑器，让我可以拖动、缩放、校准房间位置和连接关系。
```

请优先保证：

- 功能稳定；
- 数据结构清晰；

- 不破坏已有游戏流程；

## 1. 本次必须实现的内容

### 1.1 地图编辑器页面

请新增一个开发用页面，例如：

```text
/dev/map-editor
```

该页面用于编辑地图，不是正式玩家游戏页面。

页面需要包含：

- 地图编辑画布；
- 原地图参考底图；
- 房间矩形格子；
- 房间编号和房间名称；
- 连接线；
- 当前选中房间的信息面板；
- 当前选中连接的信息面板；
- 保存 / 导出 / 导入功能；
- 地图校验结果区域。

UI 不需要精美，清楚、稳定、可用即可。

---

### 1.2 根据原地图生成结构化草稿

请基于 `禁闭逃杀_地图.png` 和 `禁闭逃杀_规则手册.md` 中的房间信息，生成一份初始地图草稿。

要求：

- 每个房间用矩形格子表示；
- 格子内显示房间编号，例如 `B107`；
- 如果有房间名称，也显示名称，例如 `武器库`；
- 楼层结构、左右分布、上下关系、房间大致顺序尽量接近原图；
- 不要求像素级还原；
- 不要重新绘制美术地图；
- 不要追求最终 UI 设计。

地图编辑器的核心是让我后续能手动修正，而不是一次性自动生成完美地图。

---

### 1.3 房间位置和大小编辑

每个已有房间格子必须支持：

- 拖动位置；
- 调整宽度；
- 调整高度；
- 选中后查看当前坐标；
- 选中后查看当前尺寸；
- 修改后能保存；
- 重新打开页面后能恢复保存结果。

坐标建议使用画布坐标系统，例如：

```ts
type RoomLayout = {
  id: string;
  name: string;
  floor: string;
  x: number;
  y: number;
  width: number;
  height: number;
};
```

注意：

- 坐标只用于显示；
- 坐标不能直接作为移动规则；
- 移动规则必须依赖房间连接图结构。

---

### 1.4 原地图作为参考底图

地图编辑器必须支持显示原地图 `禁闭逃杀_地图.png` 作为参考底图。

需要支持：

- 显示底图；
- 隐藏底图；
- 调节底图透明度；
- 房间格子覆盖在底图之上；
- 连接线覆盖在底图之上。

这样我可以一边看原图，一边拖动新房间格子进行校准。

---

### 1.5 连接关系编辑

地图编辑器需要支持编辑已有房间之间的连接关系。

连接类型至少包括：

```ts
type RoomConnectionType =
  | 'adjacent'   // 普通相邻
  | 'stairs'     // 楼梯
  | 'bridge'     // 廊桥
  | 'pipe'       // 管道，例如 B105 -> B503
  | 'teleport'   // 传送
  | 'helicopter' // 停机坪直升机
  | 'special';   // 其他特殊连接
```

连接数据建议类似：

```ts
type RoomConnection = {
  id: string;
  from: string;
  to: string;
  type: RoomConnectionType;
  bidirectional: boolean;
  needReview?: boolean;
  note?: string;
};
```

必须支持：

- 查看某个房间已有连接；
- 选择两个已有房间建立连接；
- 修改连接类型；
- 修改连接是否双向；
- 删除错误连接；
- 标记某条连接 `needReview`；
- 为连接添加备注；
- 保存连接结果；
- 导出连接数据。

重点：

- 梯子、管道、廊桥要能作为连接类型被识别；
- 不确定的连接不要硬编，标记为 `needReview`；
- 后续路径计算必须用连接图，而不是用房间坐标距离。

---

### 1.6 保存、导入、导出

地图编辑器必须支持保存编辑结果。

至少支持：

- 保存当前布局；
- 重新加载已保存布局；
- 导出完整地图 JSON；
- 从 JSON 导入地图；
- 导出布局数据；
- 导出连接数据。

建议最终形成这些文件或等价结构：

```text
src/game/config/mapLayout.ts
src/game/config/mapGraph.ts
```

或者：

```text
src/game/config/mapLayout.json
src/game/config/mapGraph.json
```

要求：

- 文件结构清晰；
- 方便人工修改；
- 方便后续正式游戏读取；
- 不要把数据只存在 React state 里。

---

### 1.7 地图校验功能

请在 `/dev/map-editor` 中增加简单地图校验区域。

至少检查：

- 是否有规则手册中存在但地图数据中缺失的房间；
- 是否有地图数据中存在但规则手册中不存在的房间；
- 是否有孤立房间；
- 是否有未确认连接 `needReview`；
- 是否存在重复连接；
- 是否存在连接指向不存在房间；
- 是否存在明显单向连接但未写备注的情况。

如果方便，可以增加简单路径检查：

- 选择起点房间；
- 选择终点房间；
- 显示最短路径；
- 显示预计步数；
- 显示经过的连接类型。

这只是开发检查工具，不是正式玩家行动页。

## 3. 数据结构要求

请把显示布局和逻辑连接分开。

### 3.1 显示布局数据

负责地图看起来长什么样。

```ts
type RoomLayout = {
  id: string;
  name: string;
  floor: string;
  x: number;
  y: number;
  width: number;
  height: number;
};
```

### 3.2 逻辑连接数据

负责地图怎么走。

```ts
type RoomConnection = {
  id: string;
  from: string;
  to: string;
  type: 'adjacent' | 'stairs' | 'bridge' | 'pipe' | 'teleport' | 'helicopter' | 'special';
  bidirectional: boolean;
  needReview?: boolean;
  note?: string;
};
```

### 3.3 房间基础信息

房间基础信息可以从既有规则配置或房间配置中读取，不要在本次编辑器中额外做复杂编辑。

可以展示：

- 房间编号；
- 房间名称；
- 所属楼层。

但本次不要提供这些字段：

- 是否出生房间；
- 是否功能房间；
- 是否可抽卡房间；
- 是否普通房间。

## 4. 交互要求

### 4.1 房间拖动

- 鼠标拖动房间矩形；
- 拖动时实时更新位置；
- 松开鼠标后保存到当前编辑状态；
- 页面刷新前如果未保存，应提示或至少提供明显保存按钮。

### 4.2 房间缩放

- 选中房间后，可以修改宽高；
- 可以通过输入框修改；
- 如果实现拖拽缩放手柄更好，但不是必须；
- 宽高不能小于合理最小值。

### 4.3 连接编辑

- 选择起点房间；
- 选择终点房间；
- 选择连接类型；
- 选择是否双向；
- 可添加备注；
- 点击保存连接；
- 连接线显示在画布上。

### 4.4 底图控制

- 开关显示底图；
- 滑块调节底图透明度；
- 房间和连接始终显示在底图上方。
