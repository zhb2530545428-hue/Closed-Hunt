慈善家赠予事件需要公开赠予双方角色名。

### 2. 黑客关闭房间功能：按房间效果结算时点区分

黑客关闭房间功能不是简单地全部按“进入先后”判断，必须区分房间效果的生效时点。

#### 2.1 行动阶段 / 进入后立即处理的房间效果

对于行动阶段进入房间后立即处理的效果，例如：

- 抽卡房间抽卡；
- 基因库；
- 操作室；
- 控制室；
- 其他在玩家行动时立即触发或立即使用的房间功能；

黑客关闭房间功能只影响：

> 黑客行动之后，才进入该房间或才尝试使用该房间功能的玩家。

也就是说，如果某名玩家在黑客行动之前已经进入并完成了该房间功能，不能被黑客回溯取消。

#### 2.2 结算阶段才生效的房间效果

餐厅和手术室属于结算阶段才真正生效的房间效果。

因此如果本轮黑客在结算前关闭了餐厅或手术室，则结算阶段该房间功能统一失效，不分玩家是黑客行动前进入还是黑客行动后进入。

必须按以下规则处理：

##### B204 餐厅

如果 B204 餐厅本轮被黑客关闭：

- 结算阶段餐厅不提供免交水粮效果；
- 所有位于餐厅的存活玩家都不能因为餐厅而免除水粮；
- 玩家是否扣血仍按本轮水粮上交情况正常判断；
- 不能因为“玩家在黑客关闭前已经进入餐厅”就保留免水粮。

##### B202 手术室

如果 B202 手术室本轮被黑客关闭：

- 结算阶段不能触发手术室特殊效果；
- 2 人在手术室时不能手术加血；
- 不能因为“玩家在黑客关闭前已经进入手术室”就保留手术回血；
- 同房间存活玩家应按普通房间处理，后续战斗 / 乱斗逻辑按现有规则执行。

注意：请重点检查手术室原本可能存在的“2 人不战斗且 +4 生命”逻辑，被黑客关闭后不能继续保留该特殊保护。

#### 2.3 推荐实现方式

请不要在各个 UI 组件里零散判断黑客关闭。

建议统一封装类似工具函数：

```ts
isRoomFunctionDisabledForAction(roomId, playerActionOrder, hackerCloseActionOrder, gameState)
isRoomFunctionDisabledForResolution(roomId, gameState)
```

其中：

- `isRoomFunctionDisabledForAction` 用于行动阶段即时效果，考虑进入 / 使用时点；
- `isRoomFunctionDisabledForResolution` 用于结算阶段效果，只要该房间本轮在结算前被关闭，则结算时统一失效。

如现有代码结构不适合完全照搬函数名，可用等价结构，但必须保证逻辑清楚、集中、可测试。

### 4. 饮品师多瓶果汁限制

饮品师一次使用多瓶果汁时，同一目标不可以被使用多瓶。

要求：

- 如果使用 N 瓶果汁，需要选择 N 个目标；
- N 个目标必须互不相同；
- 如果 UI 已经支持多目标选择，不要大重构，只补充去重校验；
- 校验失败时提示玩家重新选择。

---

### 5. 暗影使者恢复修正

暗影使者的恢复量按照本轮暗影实际吸取的生命点数计算。

也就是说：

```text
本轮所有其他暗影一共实际吸到几点血，存活状态下的暗影使者就恢复几点生命。
```

不要再按“吸血事件次数”或其他错误口径计算。

恢复仍然不能超过生命上限。

---

预言家没有决定顺位能力。

- 从职业配置中移除预言家的决定顺位能力；
