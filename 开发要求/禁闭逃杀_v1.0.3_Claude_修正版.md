# 禁闭逃杀 v1.0.3 Claude 修正版开发指令

## 版本名称

`v1.0.3：信息隐藏、行动 UI、属性校验与房间关闭修复包`

> 本修正版已移除“果汁目标选择修复”专项，因为该问题已经修好。请不要重复重构果汁目标选择逻辑。

---

## 一、任务背景

当前《禁闭逃杀》电子版已经推进到 v1.0 系列，基础玩法、地图、结算、道具、职业和本地热座等功能已经基本可用。

本次任务不是重做项目，也不是新增大功能，而是根据最新试玩反馈继续修复规则漏洞、信息隐藏问题、属性校验问题、房间关闭问题和行动 UI 顺序问题。

请先阅读：

1. `禁闭逃杀_规则手册.md`
2. `禁闭逃杀_地图.png`
3. 当前代码实现

再开始修改。

## 二、本次必须修复的问题

### 1. 慈善家基因转移逻辑修复

当前问题：

慈善家赠予道具后，系统会自动让对方转出最高的基因点。

正确规则：

- 慈善家强制赠予 1 张道具给其他 1 名存活玩家。
- 被赠予玩家必须公开将 1 点基因永久转移给慈善家。
- 具体转出哪一项基因，必须由被赠予玩家自己选择。
- 不能由系统自动选择最高项。
- 不能选择当前数值为 0 的基因。

实现要求：

- 当慈善家赠予成立后，给被赠予玩家生成一个待处理选择。
- 被赠予玩家需要选择：武力 / 速度 / 负重中的一项。
- 只允许选择当前大于 0 的属性。
- 选择完成后，该属性 -1，慈善家获得 1 点对应属性。
- 相关日志要脱敏，不能泄露不该公开的信息。

---

### 2. 自由阶段交易对象显示角色名称

当前问题：

自由阶段交易选择对象时仍显示玩家 ID，阅读混乱。

修复要求：

- 交易对象下拉框 / 按钮 / 确认文案中，优先显示角色名称。
- 可在必要时附带昵称，但不要只显示玩家 ID。
- 首页或玩家个人身份确认页可以保留玩家 ID。
- 游戏正式流程中，尽量使用角色名称作为玩家识别文本。

---

### 3. 本地测试切换身份处显示角色名称

当前问题：

本地热座 / 本地测试切换身份处只显示玩家 ID。

修复要求：

- 切换身份处显示角色名称。
- 如需辅助区分，可显示：`角色名称 / 昵称`。
- 不要让测试时只能看到一堆 playerId。

### 4. 公开日志脱敏修复

只改用户明确指出有问题的部分。没有指出的问题不要大改。

#### 4.1 行动阶段完成提示不要公开

当前问题：

公开日志显示“谁已完成行动”。

修复要求：

- 行动阶段公开日志中不要显示谁完成了行动。
- 房主可以在裁判视图看到行动进度。
- 普通玩家不能通过公开日志推断行动顺序以外的隐私信息。

#### 4.2 战斗 / 乱斗日志脱敏

当前问题：

结算阶段公开日志显示谁打了谁、谁在房间里、谁扣了几点血等信息。

正确公开信息：

- 玩家只能知道哪个房间发生了战斗或乱斗。
- 可以显示：`B107 发生战斗`、`B501 发生乱斗`。
- 不能显示参战者身份。
- 不能显示谁攻击谁。
- 不能显示每个人扣了几点血。
- 不能显示谁具体在那个房间。
- 最后结算全部完成后，统一显示所有玩家最终生命值。

实现要求：

- 建立或强化统一日志脱敏层。
- 不要在各个结算函数里直接拼公开日志。
- 公开日志、玩家私密日志、房主裁判日志要区分。

#### 4.3 位置泄露类日志全面检查

必须检查所有可能暴露其他玩家位置的公开信息，包括但不限于：

- 毒气伤害
- 激光室伤害
- 火箭筒伤害
- 水粮扣血
- 暗影吸血
- 房间效果触发
- 抽卡失败
- 黑客关闭房间后的提示
- 餐厅免水粮
- 手术室回血
- 药房 / 仓库 / 水塔 / 粮仓抽卡

公开日志不得出现：

- 某玩家在哪个房间被毒气伤害
- 某玩家在哪个房间被激光伤害
- 某玩家在哪个房间没交水粮
- 某玩家在哪个房间被火箭筒炸
- 某玩家在哪个房间触发了房间效果
- 某玩家因为在餐厅所以免交水粮

允许公开：

- 哪些房间发生战斗 / 乱斗
- 哪个楼层成为毒气楼层
- 火箭筒袭击了哪个房间
- 最终统一生命值
- 哪些玩家变成暗影
- 哪些暗影将在下一轮复活

但具体中间伤害来源和位置不要暴露给其他玩家。

### 5. 属性校验修复

#### 5.1 速度永远不能为 0

修复要求：

- 初始速度不能为 0。
- 游戏过程中速度不能被降到 0。
- 如果某些效果导致速度可能为 0，必须拦截或设为最低 1。
- UI 表单也要限制。

#### 5.2 初始属性加和不能超过 10

修复要求：

- 开局基因点分配：武力 + 速度 + 负重不能超过 10。
- 如果没有分满 10，可以根据当前规则允许或提示；但绝不能超过 10。
- 提交准备前必须校验。

#### 5.3 负重为 0 时不能拿次元口袋

修复要求：

- 次元口袋本身是一张道具卡。
- 如果玩家当前负重为 0，且没有次元口袋，不能获得次元口袋。
- 不能用“拿到次元口袋后负重无限”绕过“次元口袋自身也需要被拿起”的限制。
- 抽卡、交易、赠予、回收装置、停尸间等所有获得道具路径都要检查。

---

### 6. 黑客关闭房间效果统一修复

当前问题：

黑客关闭功能房间后，UI 显示该房间被关闭，但实际效果仍可能生效。

示例：

- 餐厅被关闭后，催眠师进入餐厅，UI 显示不能触发，但实际仍免交水粮。

修复要求：

- 只要房间被黑客关闭，本轮该房间：
  - 不能抽卡；
  - 不能使用房间功能；
  - 不能产生任何功能性收益；
  - 不能触发餐厅免水粮；
  - 不能触发手术室回血；
  - 不能触发仓库 / 水塔 / 药房 / 武器库等抽卡；
  - 不能触发控制室、基因库、操作室等效果；
  - 不能触发停机坪空投收益；
  - 不能触发其他功能房间收益。

实现要求：

- 统一封装一个类似 `isRoomFunctionAvailable(roomId, gameState)` 的判断。
- 所有房间效果、抽卡、免水粮、回血、空投、基因调整等都必须经过这个判断。
- 不要每个房间零散判断，避免漏掉。
- 进入被关闭房间的玩家可以私密知道“该房间功能已被关闭”。
- 其他未进入该房间的玩家不应知道。

### 7. 水粮与行动 UI 顺序优化

#### 7.1 首轮也要询问是否为第 1 轮交水粮

用户定义的轮次表达：

- 游戏有一个“首轮”。
- 首轮之后是第 1 轮至第 6 轮。
- 不要显示成第 7 轮。
- 首轮行动时，也要询问是否为下一轮，也就是第 1 轮交水粮。

修复要求：

- 首轮也显示水粮准备按钮。
- 文案统一为：`是否为下一轮交水粮？`
- 玩家在本轮行动末尾选择是否预交下一轮水粮。
- 下一轮结算时根据上一轮的选择处理。

#### 7.2 行动 UI 顺序重排

当前 UI 问题：

玩家确认移动后，抽卡、背包更新、水粮/药/果汁/肾上腺素等按钮位置不顺，甚至需要往上翻，体验反人类。

新的行动 UI 顺序必须为：

1. 当前玩家确认身份
2. 显示当前位置、速度、可达房间
3. 选择并确认移动目标
4. 如果目标房间可抽卡，强制处理抽卡
5. 抽卡后立即显示抽到的东西
6. 背包 / 负重状态更新
7. 显示下一轮准备区：
   - 水
   - 粮
   - 药
   - 果汁
   - 肾上腺素
8. 上述按钮无论有没有道具都要显示
9. 数量为 0 时按钮禁用或提示不可用
10. 明确声明这些选择是为下一轮生效
11. 毒气投票
12. 最后才出现黄色“结束行动”按钮

特别注意：

- UI 不要在确认移动后跳回页面上方。
- 后续操作区域要顺着往下出现。
- 黄色结束按钮只能在必要流程完成后出现。

---

### 8. 本地热座模式增加单个随机生成玩家

当前问题：

本地热座测试时，只能一键生成 9 个玩家，或者手动一个个定义角色和 ID，太麻烦。

修复要求：

- 保留一键生成 9 个玩家功能。
- 新增一个“随机生成 1 名玩家”按钮。
- 点击一次，只生成 1 名玩家。
- 生成时自动分配：
  - 随机昵称 / 测试名；
  - 未被占用的角色；
  - 合法初始属性；
  - 合法出生房间；
  - 必要 ID。
- 不要生成重复角色。
- 如果已经 9 名玩家，则按钮禁用。
- 如果角色不足或无可用角色，给出提示。

## 四、实现建议

### 1. 日志必须统一脱敏

建议建立统一函数，例如：

```ts
type LogVisibility = 'public' | 'private' | 'host';

function createPublicLog(event: GameEvent): PublicLogEntry;
function createPrivateLog(event: GameEvent, viewerPlayerId: string): PrivateLogEntry;
function createHostLog(event: GameEvent): HostLogEntry;
```

禁止在结算函数里随意拼公开字符串。

### 2. 房间关闭必须统一判断

建议建立统一函数：

```ts
function isRoomFunctionAvailable(roomId: RoomId, state: GameState): boolean;
function assertRoomFunctionAvailable(roomId: RoomId, state: GameState): Result;
```

所有房间功能都要经过它。

### 3. 属性和道具获得要统一校验

建议建立统一函数：

```ts
function validateInitialGenes(force: number, speed: number, load: number): ValidationResult;
function canGainItem(player: Player, item: Item, state: GameState): ValidationResult;
function gainItem(playerId: string, item: Item, source: ItemSource): GameState;
```

不要让不同来源绕过规则。

## 五、验收标准

完成后应能通过以下鼠标验证：

1. 慈善家赠予后，被赠予玩家自己选择转出哪项基因。
2. 被赠予玩家不能选择数值为 0 的基因。
3. 自由阶段交易对象显示角色名称。
4. 本地测试切换身份显示角色名称。
5. 行动阶段公开日志不显示谁完成了行动。
6. 战斗公开日志只显示哪个房间发生战斗 / 乱斗。
7. 战斗公开日志不显示参战者、不显示扣血、不显示谁在房间。
8. 毒气、激光、火箭筒、水粮、暗影等日志不能暴露玩家具体位置。
9. 结算结束后才统一显示最终生命值。
10. 初始速度不能设为 0。
11. 游戏过程中速度不能变为 0。
12. 初始属性总和不能超过 10。
13. 负重为 0 时不能获得次元口袋。
14. 黑客关闭餐厅后，进入餐厅不能免水粮。
15. 黑客关闭抽卡房间后，不能抽卡。
16. 黑客关闭手术室后，不能回血。
17. 首轮也能选择是否为第 1 轮交水粮。
18. 行动 UI 按“移动 → 抽卡 → 背包更新 → 下一轮准备 → 毒气投票 → 结束行动”顺序展示。
19. 水、粮、药、果汁、肾上腺素按钮数量为 0 时也显示但禁用。
20. 本地热座可以随机生成 1 名玩家。
21. 随机生成 1 名玩家不会产生重复角色。
